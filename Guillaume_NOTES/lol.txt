# 🧩 Pourquoi PureBasic bloque certaines instructions ASM

PureBasic **n’accepte pas tout l’assembleur**, même si tu écris du code valide pour NASM, MASM ou FASM.

Voici les raisons principales :

---

# 🟥 1. **PB inline ASM n’est pas un assembleur complet**

PureBasic utilise **son propre mini‑assembleur interne**, pas un assembleur externe.

Donc :

- pas de directives avancées (`ORG`, `SECTION`, `BITS`, `ALIGN`, etc.)
- pas de macros
- pas de labels globaux
- pas de jumps hors du bloc ASM
- pas de gestion de segments mémoire
- pas de code 16‑bit ou 32‑bit réel mode
- pas de bootloader (car PB génère un EXE Win32, pas un binaire brut)

👉 **Tu ne peux pas écrire un bootloader dans PB**, car PB génère un PE/EXE, pas un raw binary.

---

# 🟧 2. **PB impose le mode 32‑bit ou 64‑bit selon la compilation**

Donc :

- si tu compiles en 32‑bit → ASM 32‑bit seulement  
- si tu compiles en 64‑bit → ASM 64‑bit seulement  

Et certaines instructions :

- `mov eax, cr0`
- `lgdt`
- `cli`
- `hlt`
- `jmp far`
- `int 0x10`
- `int 0x13`

…sont **interdites** en mode protégé Windows.

👉 Windows bloque les instructions “ring 0” → PB ne peut pas les assembler.

---

# 🟨 3. **Certaines instructions sont interdites par Windows**

Même si PB les assemblerait, Windows les bloque :

- instructions privilégiées (ring 0)  
- accès aux registres de contrôle  
- accès direct au BIOS  
- interruptions matérielles  
- gestion du GDT/IDT  
- accès direct au port I/O (`in`, `out`)  

Donc :

👉 **Tu ne peux pas faire du bas niveau type bootloader dans un EXE Windows.**

---

# 🟦 4. **PB inline ASM ne peut pas changer le format du binaire**

Un bootloader doit :

- commencer à l’offset 0  
- être exactement 512 bytes  
- contenir la signature `0x55AA`  
- être en mode réel 16‑bit  
- ne pas avoir d’en‑tête PE  

PureBasic génère :

- un EXE PE32/PE64  
- avec sections `.text`, `.data`, `.rdata`  
- avec un loader Windows  
- avec un point d’entrée WinMain  

Donc :

👉 **Impossible de produire un bootloader avec PB inline ASM.**

---

# 🟩 5. **Certaines instructions ASM sont bloquées par le compilateur PB lui‑même**

PureBasic refuse :

- les instructions non supportées par son assembleur interne  
- les syntaxes MASM/NASM non reconnues  
- les labels externes  
- les jumps hors du bloc ASM  
- les directives de compilation  

Exemple :  
`jmp myLabel` → fonctionne  
`jmp _global_label` → erreur  
`org 0x7C00` → erreur  
`bits 16` → erreur

# 🧩 1. Oui, le C devient de l’ASM…  
Mais **pas l’ASM que tu veux**.

Quand tu compiles du C :

- le compilateur génère de l’ASM **pour ton environnement**  
- cet ASM dépend de **l’ABI**, **l’OS**, **la libc**, **les conventions d’appel**, etc.

Donc :

👉 **C → ASM Windows**  
👉 **C → ASM Linux**  
👉 **C → ASM baremetal**  
…ce ne sont PAS les mêmes ASM.

C’est ça la nuance.

---

# 🟦 2. Un *shim* n’est pas là pour “compiler en ASM”  
Un *shim* sert à **remplacer les appels système que la lib attend**, par *tes propres versions*.

Exemple : OpenSSL veut appeler :

- `gettimeofday()`  
- `read()`  
- `write()`  
- `fopen()`  
- `socket()`  
- `getrandom()`  
- `pthread_mutex_lock()`  

En baremetal, rien de ça n’existe.

Donc tu écris un *shim* :

```c
int gettimeofday(struct timeval *tv, void *tz) {
    tv->tv_sec = kernel_time_seconds();
    tv->tv_usec = kernel_time_microseconds();
    return 0;
}
```

Tu remplaces l’appel système par **ta version maison**.

C’est ça un shim.

---

# 🟥 3. Pourquoi tu en as besoin en baremetal  
Parce que même si ton C devient de l’ASM, **OpenSSL n’est pas écrit pour tourner sans OS**.

Il s’attend à :

- une libc  
- un système de fichiers  
- un système de threads  
- un système de sockets  
- une source d’entropie  
- un timer  
- un scheduler  
- un environnement POSIX minimal

Donc tu dois “mentir” à OpenSSL en lui donnant **des fonctions compatibles**, même si derrière c’est ton kernel qui les gère.

C’est ça le rôle des shims.

---

# 🟩 4. Exemple concret : OpenSSL veut lire un fichier PEM  
OpenSSL appelle :

```c
fopen("cert.pem", "r");
```

En baremetal, tu n’as pas `fopen`.

Donc tu fais :

```c
FILE *fopen(const char *path, const char *mode) {
    return fat32_open(path, mode);
}
```

Ton shim redirige vers ton FAT32 maison.

Et là, soudainement…  
👉 OpenSSL fonctionne en baremetal.  
👉 Parce que tu lui as donné ce qu’il attendait.
